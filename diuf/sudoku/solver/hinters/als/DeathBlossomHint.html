Stem : {0}
ALSs : {2}
Regardless of the correct value of the stem cell {0}, one of these Almost Locked Sets (ALSs) will become a Locked Set (LS), and ALL of these Locked Sets preclude the same candidate/s.

Hence we can eliminate those common candidates/s without knowing which ALS will become an LS.

Therefore we can remove {1}.

Explanation from https://www.sudopedia.org/wiki

A Death Blossom consists of a "stem" cell and an Almost Locked Set (or ALS) foreach of the stem cell's candidates. The ALS associated with a particular candidate of the stem cell has that value as one of its own candidates, and within the ALS, every cell that has the value as a candidate can see the stem cell. The ALSs can't overlap; ie no cell can belong to more than one ALS. Also, there must be at least one value that is a candidate of every ALS, but is not a candidate of the stem cell. (1) These are the value/s (usually one) to eliminate.

Having found a Death Blossom, if an outside cell (not in any of the ALSs) sees all occurrences of that value (in all ALSs and not in the stem cell) in the Death Blossom then we can eliminate that value from the outside cell.

(1) Each value-to-eliminate must be common to all ALSs in the DeathBlossom and not a potential value of the stem cell; which makes sense: the main logic is one of the ALSs will become an LS, so all ALSs must share that nonstem-value, precluding it from external cell/s seeing all instances in the DeathBlossom.

There's nothing to stop a DeathBlossom eliminating multiple values, it's just rarer than rocking horse s__t. In fact it never happens in top1465, but SE handles multiples anyway, just in case, but it's untested, so caveat emptor.

NOTE: SE's DeathBlossom implementation examines stems with 2 or 3 maybes, because DB is slow and top1465 doesn't have any DB's on 4 values, but I believe that 4's do exist, so the rest of the code handles them.
